# 云数据库第一阶段接入计划

更新时间：2026-06-21

## 当前判断

现在网站已经能统一入口、登录、按权限显示系统，也有本机整站备份能力。下一步不应该先改所有业务页面，而应该先把通用底座接到云端。

第一阶段只做这些：

- 员工账号
- 权限
- 操作日志
- 整站备份记录
- 系统配置
- 本机备份导入云端的中间记录

## 为什么先做这些

这些数据是所有系统共用的。先把它们放到云端，后面排课、财务、招生、教学质量等模块迁移时，就能共用同一套账号、权限、日志和备份机制。

如果先迁移某个业务系统，后面还要回头补账号权限和日志，会更乱。

## 第一阶段交付标准

完成后应达到：

- 员工账号不再只写在前端 JS 文件里。
- 管理员新增员工后，其他电脑登录也能看到。
- 权限修改后，所有设备立即生效。
- 每次新增、修改、删除、导入、导出都有云端操作日志。
- 管理员导出整站备份时，云端记录备份时间、操作人、数据类型和校验信息。
- 仍保留本机 JSON 导出，作为云端之外的兜底。

## 建议实施顺序

1. 建云数据库表结构。
2. 写云端 API：登录、读取员工、读取权限、写操作日志、记录备份。
3. 把 `portal/auth.js` 的员工账号读取改成优先读云端，失败时再用本地内置名单兜底。
4. 把 `portal/business-modules.js` 的操作日志改成优先写云端，失败时继续写本机日志。
5. 把 `portal/data-sync.js` 的备份导出增加云端登记。
6. 连续试用 3 到 5 天，确认登录、权限、日志稳定。
7. 再开始迁移建议系统、招生线索、教学质量等低风险业务数据。

## 仍然暂不迁移的内容

第一阶段先不迁移：

- 排课正式数据
- 财务正式结算数据
- 学生档案正式主数据
- 课时费和分红核算主数据

这些数据强关联、风险高，要等字段和人工 Excel 对齐后再迁。

## 对老师使用的影响

第一阶段接云数据库后，老师看到的页面变化应该很小。

主要变化是：

- 登录更稳定。
- 权限跨设备同步。
- 管理员能追溯谁改了什么。
- 不同电脑之间基础账号和权限不再各自为政。

业务录入方式暂时不大改，避免老师下周试用时突然不适应。
