## 2. 桌面统一入口建议

不建议把源码整体搬到桌面。

原因：这个项目已经接入 Git、Next 静态构建、GitHub Pages 发布路径和多个静态资源目录。直接搬动源码位置，容易造成构建路径、发布路径、脚本引用混乱。

推荐做法：在桌面建立一个“匠人程系统总入口”文件夹，里面放：

- 本总索引文件
- 指向真实源码目录的快捷方式
- 常用线上网址说明
- 每日备份说明

这样老师和管理员容易找到入口，同时源码仍然保留在稳定位置。

## 5. 当前数据状态

当前大部分业务系统仍是轻量化前端版本，核心数据主要存在当前浏览器的 `localStorage`。

这意味着：

- 同一台电脑、同一个浏览器能继续看到录入的数据。
- 换电脑、换手机、换浏览器，数据不会自动同步。
- 多人同时录入时，暂时不能做到真正云端实时统一。
- 正式接云数据库前，必须每天导出整站备份。

排课系统当前已纳入整站备份的旧版数据：

- 平时模式数据：`paike-june-system-v1`
- 平时模式状态：`paike-june-system-meta-v1`
- 暑假模式数据：`paike-system-prototype-v1`
- 暑假模式状态：`paike-system-prototype-meta-v1`
- 暑假待确认项：`paike-summer-import-review-v1`
- 排课系统数据同步索引：`jrc-paike-legacy-cloud-transition-v1`

下周试用期的临时要求：

- 每天下班前由管理员进入“数据同步与备份中心”
- 导出一次整站备份
- 备份文件按日期保存
- 遇到系统异常、浏览器清理、换电脑前，也要先导出备份

## 6. 云数据库接入节奏

云数据库正式方向改为阿里云。原因是国内长期使用、付款、控制台维护和访问稳定性更适合公司正式系统。

当前已准备好的阿里云落地包：

- 购买配置说明：`deploy/aliyun/README.md`
- 第一阶段 API 规格：`deploy/aliyun/api-contract.md`
- 数据库表结构：`database/cloud-schema-v1.sql`
- 员工与权限初始化：`deploy/aliyun/seed-employees.sql`
- 后端 API 骨架：`deploy/aliyun/api/server.mjs`

云数据库不是不接，而是按顺序接，避免把还没稳定的业务规则一次性写死。

阶段 0：当前阶段

- 保持现有前端系统可试用
- 每天导出整站备份
- 继续打磨字段、权限、Excel 导入、排课规则、财务规则

阶段 1：先接基础安全与通用底座

- 员工账号
- 登录会话
- 权限表
- 操作日志
- 备份记录
- 系统配置

阶段 2：先迁移低风险业务数据

- 建议管理系统
- 招生线索
- 招生跟进记录
- 试听预约与试听反馈
- 教学质量抽样巡课和问卷记录

阶段 3：再迁移强关联核心业务

- 学生档案
- 班级档案
- 排课数据
- 教室占用
- 调课、停课、补课记录
- 财务支出、课时费、分红、提成核算

阶段 4：正式使用前完成

- 云端自动备份
- 权限细分到查看、新增、编辑、删除、导出
- 操作日志可追溯
- Excel 批量导入
- 数据字段统一
- 手机端录入体验优化

接云数据库和批量导入 Excel 前，统一按 `docs/data-field-standards.md` 里的字段规范执行。老师继续用 Excel 也可以，但姓名、手机号、日期、班级名、金额这些关键字段要逐步往模板靠。

## 7. 为什么不一开始直接全量接云数据库

不是因为技术上不能做，而是因为当前业务规则还在和老师的人工 Excel 工作方式磨合。

如果一开始直接全量接入云数据库，风险是：

- 字段没定型，后面改表会频繁迁移数据。
- 权限没完全定，容易出现该看不到的人看到敏感信息。
- 排课、财务、招生之间的联动规则还在确认，过早固化容易算错。
- 老师还在并行用 Excel，云端数据和人工表容易出现两套标准。

更稳的做法是先让系统能用、能备份、能比对，再把稳定的数据模块逐个迁到云端。

## 8. 后续工作原则

- 不换网址，继续使用 `https://jrc-edu.github.io/jrcedu/`
- 首页保留统一工作台入口
- 各系统保持独立入口，但共用账号、权限、数据底座
- 先保证数据准确，再做自动化
- 先保证老师容易上手，再做复杂管理能力
- 重要数据进入云数据库前，必须保留人工 Excel 或导出备份作为兜底
